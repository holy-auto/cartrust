# トヨタ PoC 実施計画

> 作成日: 2026年4月26日
> 前提ドキュメント: `docs/toyota-negotiation-strategy.md`, `docs/vehicle-passport-design.md`

---

## PoC の目的と「成功した状態」

目的は **「技術が動くことを見せる」ではなく「トヨタ担当者が社内で話を通せる材料を作る」こと**。

成功した状態 =
- トヨタ担当者が「この車はLedraでコーティング施工済み、Polygonで証明されている」という画面を上司に見せられる
- 「PoC期間中に X 台の車両パスポートが発行された」という数字を持ち帰れる
- 「外部のスタートアップが既にここまで動いている」という事実で社内稟議が通る

---

## 現在のギャップと優先実装タスク

PoCを始める前に埋めなければならない技術的なギャップ。

| タスク | 状態 | 優先度 | 工数目安 |
|--------|------|--------|---------|
| `/v/[vin]` 車両パスポートページ | **未実装** | 最高 | 3〜5日 |
| アンカー成功時の passport upsert | **未実装** | 高 | 1日 |
| Polygonアンカリングのデフォルトをtrue化 | 環境変数1行 | 高 | 即時 |
| NFCのデフォルト書き込み先を `/v/{vin}` に切替 | ハードコード修正 | 中 | 半日 |
| Toyota PoC 専用デモテナント | スクリプトあり | 中 | 1時間 |

**最重要**: `/v/[vin]` ページがないとPoCのクライマックス（「この車両の全施工履歴がここに集約されている」）が見せられない。ここが最初の実装課題。

### `/v/[vin]` 実装の具体的な手順

`docs/vehicle-passport-design.md` のPR-1〜PR-4に沿って進める。

```
PR-1: DB マイグレーション（20260424000000_vehicle_passports.sql）
  └─ 状態: 完了済み（migration ファイル確認）

PR-2: アンカー成功フックで passport upsert
  └─ 場所: src/lib/anchoring/providers/polygon.ts のアンカー成功ハンドラ
  └─ 処理: upsertVehiclePassport(vin) を呼ぶ
  └─ 状態: 未実装

PR-3: /v/[vin] ページ + PassportTimeline コンポーネント
  └─ 場所: src/app/v/[vin]/page.tsx
  └─ 状態: 未実装（最大のギャップ）

PR-4: /c/[public_id] へのパスポートバッジ追加
  └─ 場所: 既存の公開証明書ページ
  └─ 状態: 未実装
```

## PoC の 3 フェーズ

### フェーズ 0: 社内準備（〜2週間、トヨタ接触前）

**目標**: 「今すぐデモできる状態」を作る。

#### Step 0-1: Polygonアンカリングを本番環境で有効化

```bash
# .env.local（またはVercel環境変数）
POLYGON_ANCHOR_ENABLED=true
POLYGON_NETWORK=polygon        # Mainnet（または amoy でテスト）
POLYGON_RPC_URL=...
POLYGON_PRIVATE_KEY=...
POLYGON_CONTRACT_ADDRESS=...
```

詳細は `docs/polygon-anchoring-deployment.md` 参照。Mainnetへの投入前に Amoy testnet で動作確認してから切り替える。

#### Step 0-2: 車両パスポートの実装（PR-2〜PR-4）

実装順序:
1. `src/lib/passport/upsertVehiclePassport.ts` — VINに紐づくpassportをupsertする関数
2. `src/lib/anchoring/providers/polygon.ts` のアンカー成功ハンドラから呼び出し
3. `src/app/v/[vin]/page.tsx` — 公開パスポートページ
4. `/c/[public_id]` ページへの「この車両の全履歴を見る →」バッジ追加

**デモで見せる最小限のUI（`/v/[vin]`）**:
- 車両情報（メーカー・車種・年式）
- サマリバッジ「アンカー済み証明書 N件 / 関与施工店 M店」
- 施工タイムライン（施工店名・施工種別・日付・Polygon Tx リンク）
- 「Polygon PoSネットワーク上で検証可能」フッター

MVPとして凝ったデザインは不要。動いて数字が出ることが最優先。

#### Step 0-3: Toyota PoC 専用デモテナントの作成

```bash
# 既存スクリプトを活用
npx tsx scripts/setup-demo-tenant.ts
npx tsx scripts/setup-demo-tenant-user.ts
```

その後、デモテナントの表示名・ロゴを「Woven City モビリティセンター（デモ）」相当に変更。車両データに実際のトヨタ車種・VINフォーマットを反映させる。

#### Step 0-4: 3分デモ動画の撮影

シナリオ:
1. 施工店側（admin）で車両・施工内容を入力、証明書を発行
2. Polygon Tx がPolygonoscanに記録される様子
3. `/v/{vin}` を開いて車両パスポートページが表示される
4. NFCタグをスキャンして同じページに到達する

この動画をメール・Slack・会議冒頭で流す。言語説明なしで伝わることが条件。

### フェーズ 1: トヨタへの初回接触とPoC合意（〜1ヶ月）

**目標**: 特定の担当者と「やってみよう」という合意を取る。

#### Step 1-1: 接触先の絞り込み

最初に狙うべき窓口（優先順）:

| 優先度 | 接触先 | 理由 |
|--------|--------|------|
| ★★★ | Woven by Toyota — スタートアップ連携担当 | Woven Cityはスタートアップとの共創を明示している。意思決定が比較的速い |
| ★★☆ | KINTO — プロダクト/事業開発 | SBT実証実験をやった部門。技術感度が高い |
| ★☆☆ | Toyota Blockchain Lab | 窓口が公式ページのみ。層が厚く時間がかかる可能性 |

**最初のメッセージ**: 提案書や長文メールは送らない。「15分だけ話せますか」から始める。LinkedInのDMで十分。

#### Step 1-2: 初回ミーティング（15〜30分）の設計

話すこと（この順番で）:
1. 「コーティング・PPFの施工証明をブロックチェーンで発行するプラットフォームです」（10秒）
2. デモ動画を共有（3分）
3. 「Woven Cityの整備施設で使ってもらえませんか。費用はLedra持ちで」（1分）

**話さないこと**: 技術スタック、契約形態、将来展望、料金体系。全部「次回」で良い。

#### Step 1-3: PoC合意書（NDA + PoC範囲書）

大企業側が必要な書類は用意する。ただし複雑にしない。

```
PoC合意書に含める内容（最小限）:
1. PoC期間: 3ヶ月
2. 対象: [施設名]での施工証明書発行
3. 費用: Ledraサブスクリプション費用は無償提供
4. データ: PoC期間中のデータはLedra/トヨタ双方が分析可能
5. 秘密保持: 相互NDA
6. 本番採用の義務はなし
```

「本番採用の義務はなし」を明記することで法務の承認が速くなる。

### フェーズ 2: PoC 実施（3ヶ月）

**目標**: 実データで「Ledra証明書がある車両」を作る。

#### PoC の実際の運用フロー

```
トヨタ側（施設スタッフ）          Ledra側
┌─────────────────────────┐    ┌────────────────────────┐
│ 1. 施工完了後にスマホで   │    │ 2. 証明書生成          │
│    Ledra管理画面を開く    │───▶│    + C2PA署名          │
│                          │    │    + SHA-256ハッシュ    │
│ 3. QRコードを印刷/送付   │◀───│ 4. Polygon アンカリング│
│                          │    │    (Tx hash 記録)       │
│ 5. VINをスキャン or 入力 │    │                        │
│    → /v/{vin} を確認     │    │ 6. 車両パスポート更新   │
└─────────────────────────┘    └────────────────────────┘
```

**トヨタ側に覚えてもらうことは1つだけ**: 「施工が終わったらLedraで証明書を発行する」。複雑な操作は不要。

#### PoC 期間中にLedraがやること

- 週次でデータレポートを送付（発行件数・アンカー成功率・パスポートアクセス数）
- 問題があれば24時間以内に対応
- 中間（1.5ヶ月時点）でミーティングを設定し、フィードバックを収集
- 追加機能の要望があれば「PoC中は最優先で対応する」姿勢を見せる

---

### フェーズ 3: 評価と本番交渉（PoC終了後）

**目標**: PoC の数字を武器に本番採用・拡大の話をする。

#### 提示する成果指標

| 指標 | 目標値 | 意味 |
|------|--------|------|
| 発行証明書数 | 50件以上 | 「実際に使われた」証拠 |
| Polygonアンカー成功率 | 95%以上 | システムの信頼性 |
| 車両パスポート発行VIN数 | 30台以上 | 「Ledra証明付きの車」が存在する |
| パスポートページ閲覧数 | 延べ100回以上 | 実際に参照された回数 |
| 施設スタッフの継続利用率 | PoC3ヶ月間継続 | 離脱なく使い続けた事実 |

#### 本番交渉での提案

PoC結果を見せた上で:

**オプションA: Woven City 全施設への展開**
- 費用: 月額 X 万円（施設数 × 単価）
- Ledra の「Woven City 公認施工証明プラットフォーム」としてのマーケティング活用を条件に価格優遇

**オプションB: TBL エコシステム統合**
- PoCデータとAPIを持参して TBL チームへの橋渡しを依頼
- Woven担当者を「社内スポンサー」として活用

**オプションC: ディーラー展開の橋渡し**
- Woven Cityのデータを実績として、トヨタ販売店への紹介を依頼

## PoC で使うデモシナリオ（台本）

### シナリオ: 「この車のコーティングは本物です」

**登場人物**:
- 施工スタッフ（Ledra admin を操作）
- 車両オーナー（QRコードを受け取る）
- 第三者（保険会社・中古車査定担当者）

**流れ**:

```
[施工スタッフ]
1. /admin/certificates/new を開く
2. 車両VINを入力（例: JT2BF22K1W0066983）
3. 施工内容「セラミックコーティング施工」と写真を登録
4. 「証明書を発行」をクリック
   → QRコード付きPDFが生成される
   → バックグラウンドでPolygon Txが発行される

[車両オーナー]
5. QRコードをスマホでスキャン
   → /c/{public_id} が開く（施工証明書）
   → 「この車両の全履歴を見る →」バッジをタップ
   → /v/{vin} が開く（車両パスポート）

[第三者（保険会社・査定担当者）]
6. /v/{vin} で確認できること:
   - 「コーティング施工: 2026-04-26 | Woven Cityモビリティセンター」
   - 「Polygon Tx: 0x... (Polygonscanで検証可能)」
   - 他にも施工記録があれば時系列で一覧表示
```

**デモのポイント**: ステップ6で「第三者が誰でもこのURLにアクセスして改ざんを検知できる」ことを強調。これがトヨタが今持っていないものです、と伝える。

## リスクと対策

| リスク | 対策 |
|--------|------|
| Polygonのガス代が高騰して Tx が詰まる | Amoy testnetをPoC中は使用し、本番契約時にMainnetに切り替える選択肢を持つ |
| トヨタ側のスタッフがLedraを使ってくれない | 操作手順を1枚のA4に収める。最初の1回はLedraスタッフが同席してオンボーディング |
| トヨタ法務がデータ管理に懸念を持つ | ブロックチェーンにはハッシュのみでPIIはオフチェーンという説明資料を先に用意する |
| PoC後に「継続見送り」になる | 「なぜ見送りか」のフィードバックを必ず取る。その理由が次のプロダクト改善の最重要インプットになる |
| 担当者が異動する | 最初から2〜3名との関係を作る。1名依存にしない |
